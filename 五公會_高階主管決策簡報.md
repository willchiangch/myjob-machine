# 五公會會計師模組專案｜高階決策簡報

> **日期**：2026/04/26 ｜ **專案**：凌群服務雲 — 會計師公會
> **本文目的**：以最精簡的篇幅呈現兩個方案的關鍵差異，供長官拍板

## 一、局勢一句話

自 2025 年 1 月起，已投入開發 **1,500 萬（102 人月）**，公會相關收入 **180 萬**，累計虧損 **1,320 萬**。MVP 已上線，五公會已同意承租但**尚未付款**。合約是「**租賃**」非「開發」，只列 8 大模組名稱、無具體功能規格。年租金僅 59.2 萬，**回本邏輯在維運合約 + CR 長尾收入**。

## 二、方案 A vs 方案 B 總覽

## 三、選了之後要做什麼（Next Steps）

### 選方案 A 必須同步做
1. **立即簽維運合約**（每會 1.2–1.6 萬/月，或年度顧問點數 60–80 萬）
2. **搬遷時修 CRITICAL bug**（取號 race condition、排程重複）→ 寫成「系統升級報告」展示價值
3. **產出舊系統技術健檢報告**，保護未來保固責任歸屬
4. **確認 UAT 定義**：搬遷完成＋資料驗證 = UAT，合約沒有功能規格

> 📎 搬遷技術細節見 → [五公會整合計畫](五公會整合計畫.md)

### 選方案 B 必須面對
1. **向長官申請追加預算 ~291 萬**（已虧損 1,320 萬基礎上）
2. **Data Mapping 立即啟動估算**（目前是零）
3. **與客戶重談時程**，6/30 不可能，至少延到 12 月
4. **舊系統維護合約不能停**，直到新系統上線
5. **五公會資料同步（36 人天）需提前 SA**，否則 scope 會爆

## 四、向五公會提案說詞（方案 A 客戶溝通要點）

> **背景：** 4/22、4/29 教學後觀察到會務人員與部分長官對新介面與新流程的抗拒程度高；習慣舊系統的會員反彈預期更強。為避免移轉計畫讓公會對外受到指責，以「舊系統架構整併搬遷至凌群服務雲」作為主訴求。

### 對客戶的核心訴求

> 「一樣是搬遷至凌群服務雲，但我們改為把現有系統調整架構，而功能與介面流程原封不動整併搬進來——合約標的完全不變，公會節費目標達成，會員完全不需要換介面。」

### 對公會的好處（依說服優先序）

| # | 說法重點 | 備註 |
|:--|:--------|:-----|
| 1 | **合約金額不變，確實達到節費目標** | 年租金不動，GCP 費用從 $2,800 降至 ~$1,060/月，公會帳面受益清楚 |
| 2 | **合約標的沒有變化，長官不受質疑** | 租賃合約只列 8 大模組名稱，舊系統即已具備，驗收風險極低 |
| 3 | **會員不需適應新介面、新流程** | 搬遷後操作方式完全相同，僅需驗收功能正常，不會引發會員投訴 |
| 4 | **網域不換、App 不需重新下載** | Android / iOS / React Native 三個 App 無需重新發版 |
| 5 | **未來更新不再需要停機維護** | Cloud Run 架構支援零停機部署（Rolling Update） |
| 6 | **大型檔案上傳問題一併解決** | GCS Signed URL 架構取代舊有 32MB 限制，PDF / 壓縮檔正常上傳 |
| 7 | **直接簽維運服務合約，不再為點數制煩惱** | 月費制度明確，預算好編列 |
| 8 | **台中公會理事長在意的五公會單一帳號登入成為可能** ⭐ | 整合後 Redis + JWT 架構為跨組織 SSO 鋪路；**需先向主管機關確認可行性，確認前作為籌碼保留，不主動承諾** |

### 已知缺點（坦誠說明，建立信任）

| 缺點 | 說明 |
|:----|:-----|
| 無新功能（如新公文會辦機制） | 本方案為「搬遷」，非「開發」；後續功能需另簽 CR |
| SSL 憑證與 SendGrid 費用仍由公會自行負擔 | 日後可改為 GCP 代管憑證（免費、自動續約）；作為後續升級誘因保留，切換前不主動承諾 |
| 現有問題的優化（如報表效能）需另行 CR | 搬遷本身已修復取號競態等 CRITICAL bug，其餘優化清單可附上作為後續合作基礎 |
